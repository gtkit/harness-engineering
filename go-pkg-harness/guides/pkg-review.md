# 包级代码审查清单

完成代码后逐条自审。不通过则修复。

---

## 1. API 设计

- [ ] 导出的 API 是否最小化？（不该导出的没导出）
- [ ] 命名是否清晰？（不重复包名、动词开头的函数、-er 后缀接口）
- [ ] 构造函数是否用 Functional Options？必选参数放签名，可选放 Option
- [ ] Option 函数内是否做了防御校验？（nil、负数、零值）
- [ ] 默认值是否合理？不传 Option 也能正常工作？
- [ ] 新增导出 API 是否有明确用户价值，不是暴露一次性内部实现？
- [ ] 新增接口 / Option / helper 是否满足抽象准入条件？

## 2. 错误处理

- [ ] 错误消息以包名开头？（`"pkgname: xxx"`）
- [ ] 可判断的错误用 sentinel error？（`var ErrXxx = errors.New(...)`）
- [ ] 需要详情的错误用自定义类型？（实现 `error` 接口）
- [ ] wrapping 用 `%w`？
- [ ] 绝无 panic？（`Must` 变体有 GoDoc 说明）
- [ ] 不暴露内部依赖的错误类型？

## 3. 文档

- [ ] 每个导出的 type/func/const/var 有 GoDoc 注释？
- [ ] 包级文档（doc.go）有用法概述？
- [ ] 核心 API 有 Example 测试？（`// Output:` 可验证）
- [ ] 并发安全性已标注？
- [ ] README 有安装、快速上手、API 概览？

## 4. 测试

- [ ] 覆盖率 ≥ 80%？
- [ ] table-driven 风格？
- [ ] 覆盖 success + error + edge case？
- [ ] Example 测试有 `// Output:`？
- [ ] 性能关键路径有 Benchmark（`b.ReportAllocs()`）？
- [ ] `-race` 通过？

## 5. 依赖

- [ ] 零外部依赖或最小依赖？
- [ ] 第三方包选型是否遵循：标准库 → gtkit 原生包 → 业界事实标准 → 其他第三方？
- [ ] gtkit 同名包若仅是事实标准库轻封装，是否评估过直连事实标准库？
- [ ] JSON 用的是 `gtkit/json` 或 `gtkit/json/v2`？（禁止 `encoding/json`）
- [ ] 没有引入比自身更重的依赖？
- [ ] `internal/` 隔离了实现细节？

## 5.1 代码质量

- [ ] 无无意义重复代码；重复逻辑已提取或说明保留原因？
- [ ] 优先复用已有内部 helper、Option、error helper、测试 fixture 和标准库能力？
- [ ] 导出 API、内部实现、错误体系、测试辅助职责清晰？
- [ ] 测试辅助没有污染生产代码，内部实现没有扩大导出面积？
- [ ] 错误路径、nil、边界输入、兼容性、性能回退均已检查？

## 5.2 可观测性

- [ ] 错误是否带足够上下文，便于调用方定位问题？
- [ ] 导出 API 是否能让调用方判断失败类型？
- [ ] 关键失败路径是否保留了足够的排障线索？

## 6. 并发安全

- [ ] 需要线程安全的类型加了锁？
- [ ] 不需要线程安全的没有多余的锁？
- [ ] GoDoc 明确标注了并发安全性？
- [ ] Benchmark 包含并行场景？

## 7. 兼容性

- [ ] 导出 API 没有破坏已有签名？（如有，需升大版本）
- [ ] 废弃 API 用 `// Deprecated:` 标记？
- [ ] go.mod 的 module path 与版本一致？（v2+ 需要 `/v2`）
- [ ] 破坏性变更是否说明迁移路径或前向兼容策略？

## 8. Go 1.26 现代特性

- [ ] 用 `slices`/`maps`/`cmp` 替代手写循环？
- [ ] 合理使用泛型（不过度、不不足）？
- [ ] 用 `b.Loop()` 替代 `for i := 0; i < b.N; i++`？
- [ ] 用 `slog` 替代 `log`（如需日志）？

## 9. 反编造

- [ ] 所有标准库 API 调用真实存在？
- [ ] 泛型约束语法正确？
- [ ] 不确定的已标注"需确认"？
