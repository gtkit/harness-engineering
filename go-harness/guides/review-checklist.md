# AI Code Review Checklist（推理型传感器）

完成代码后，**必须**按以下维度逐条自审。
任何一项不通过，必须修复后才能交付。不适用的标注 N/A。

---

## 1. 正确性

- [ ] 业务逻辑是否覆盖了所有分支和边界条件？
- [ ] 并发场景下是否有 data race？（共享变量是否加锁？channel 是否正确使用？）
- [ ] 错误路径是否都有合理的处理和返回？（没有 `_ = err`）
- [ ] nil 指针检查是否充分？（函数参数、数据库返回、外部调用返回）
- [ ] 循环是否有正确的终止条件？
- [ ] 类型断言是否使用了 comma-ok 模式？

## 2. 安全性

- [ ] 用户输入是否做了校验和 sanitize？
- [ ] 用户输入是否有长度限制？（防止超大 payload）
- [ ] SQL 查询是否使用了参数化？（无字符串拼接 SQL）
- [ ] 敏感信息是否从配置/环境变量获取？（无硬编码密钥）
- [ ] 是否有未授权访问的风险？（接口是否走了 auth 中间件）
- [ ] 日志中是否避免了打印敏感信息？（密码、token、完整请求体）
- [ ] 文件上传是否校验了类型和大小？
- [ ] 接口权限是否最小化？

## 3. 性能

- [ ] 是否有不必要的数据库查询？（N+1 查询、循环内查库）
- [ ] 列表查询是否有 LIMIT？（默认最大 100）
- [ ] 是否有 goroutine 泄漏风险？（缺少 ctx.Done 检查、缺少退出机制、缺少 defer cancel）
- [ ] 大批量操作是否分批处理？（单次 ≤ 500 条）
- [ ] 是否有不必要的内存分配？（循环内创建大 slice、不必要的 string 转换）
- [ ] 外部调用是否设置了合理超时？
- [ ] 分页 offset 过大时是否考虑游标分页？

## 4. 可维护性

- [ ] 命名是否清晰？（函数名说明做什么，变量名说明是什么）
- [ ] 是否遵循分层架构规则？（handler 没有 import gorm、service 没有 import gin）
- [ ] 是否有重复代码可以提取？
- [ ] 错误消息是否对排查有帮助？（包含上下文信息：`"create user: %w"`）
- [ ] 关键设计决策是否有注释说明 why？
- [ ] 是否符合 Go 1.26.2 现代写法？（slices/maps/cmp 等标准库新包）

## 5. 测试

- [ ] 核心业务逻辑是否有单元测试？
- [ ] 测试覆盖 success + error + edge case 三类？
- [ ] 测试使用 table-driven 风格？
- [ ] mock 通过接口注入而非 monkey patch？
- [ ] 测试断言具体？（`assert.ErrorIs` 而非 `assert.Error`）

## 6. 资源泄漏（线上高危）

- [ ] HTTP resp.Body 是否 `defer Close()`？
- [ ] 数据库事务是否在所有路径上 commit 或 rollback？
- [ ] goroutine 是否有退出机制？
- [ ] 文件句柄是否 `defer Close()`？
- [ ] context 的 cancel 是否 `defer cancel()`？

## 7. 数据一致性

- [ ] 跨表操作是否在同一事务中？
- [ ] 缓存与 DB 是否有一致性策略？
- [ ] 并发写入是否有乐观锁或悲观锁？

## 8. 线上稳定性

- [ ] panic 是否被 recovery 中间件兜住？
- [ ] 外部依赖不可用时是否有降级方案？
- [ ] 是否有合理限流？
- [ ] 日志级别是否合理？（循环内不打 Info，error 时打 Error）

## 9. 大模型相关（如涉及 LLM 调用）

- [ ] LLM 调用是否设置了合理超时？（非流式 30-60s，流式首 chunk 15s）
- [ ] 是否有重试和降级机制？
- [ ] Token 消耗是否有监控和限额？
- [ ] Prompt 是否从模板加载而非硬编码？
- [ ] 日志是否避免了打印完整的 prompt/response？
- [ ] 流式调用是否正确处理了 ctx 取消和错误 chunk？

## 10. 支付相关（如涉及支付）

- [ ] 金额是否使用 int64（分）而非 float？
- [ ] 是否实现了幂等性？
- [ ] 回调是否验签？
- [ ] 状态流转是否单向不可逆？
- [ ] 是否先落库再调支付网关？
- [ ] 是否有超时关单和主动查询补偿？

## 11. 交叉验证

- [ ] 代码 ↔ 需求：每个功能点已实现，无多余功能
- [ ] 代码 ↔ 测试：每个公开方法有对应测试
- [ ] 代码 ↔ 文档：GoDoc 注释与实际行为一致
- [ ] 新代码 ↔ 存量代码：不破坏已有测试，风格一致

## 12. 反编造检查

- [ ] 所有引用的包名、函数名、结构体字段都真实存在？
- [ ] 没有杜撰不存在的 API 或参数？
- [ ] 不确定的地方是否已标注"需确认"？

---

## 合规摘要模板

每次交付代码时，在末尾附上：

```
## 合规检查摘要
- [x] Go 版本：1.26.2 现代特性已使用
- [x] 分层架构：符合
- [x] 错误处理：所有 error 已处理
- [x] 并发安全：无共享状态 / 已加锁
- [x] 超时控制：外部调用已设置 timeout
- [x] 敏感信息：无硬编码
- [x] 资源泄漏：已检查
- [x] 测试：核心路径已覆盖（success + error + edge）
- [x] 交叉验证：需求 ↔ 代码 ↔ 测试 已核对
- [x] 无编造内容
```
