# Architecture Guide

## 目标

本项目默认采用“可测试、可替换、可审查、可部署”的企业级分层结构，避免把业务编排、平台细节、AI 运行时、Linux 系统交互混在同一层。

## 参考目录结构

```text
.
├── apps/                  # 可执行入口，只做装配、参数解析、生命周期启动
├── services/              # 用例编排、任务流、请求协同、资源策略
├── ports/                 # 抽象接口，定义对外依赖边界
├── adapters/              # AI runtime、RPC、存储、消息、设备接入实现
├── platform/              # Linux 进程、线程、FD、信号、时钟、日志、指标等平台能力
├── libs/                  # 纯库：算法、数据结构、领域逻辑、公共工具
├── configs/               # 配置模板、样例配置、部署配置
├── models/                # 模型制品目录或模型清单
├── tests/                 # unit / integration / e2e / stress / benchmark
├── cmake/                 # CMake 模块
└── third_party/           # 仅在项目明确采用 vendor 方式时使用
```

## 依赖方向

1. `apps` 可以依赖 `services`、`ports`、`adapters`、`platform`、`libs`
2. `services` 可以依赖 `ports`、`libs`
3. `adapters` 可以依赖 `ports`、`platform`、`libs`
4. `platform` 可以依赖 `libs`
5. `libs` 不得依赖 `adapters`、`apps`
6. `tests` 可以依赖任何被测模块，但测试专用工具不得反向污染生产模块

## 强制边界

1. `apps`
   - 不写业务逻辑
   - 不写复杂资源策略
   - 只负责装配、启动、停止、信号接管、配置装载
2. `services`
   - 不直接调用 Linux 底层细节
   - 不直接硬编码 AI Runtime 细节
   - 依赖抽象接口而不是具体实现
3. `ports`
   - 定义接口、契约、错误语义、超时语义
   - 不放实现
4. `adapters`
   - 对接具体模型后端、设备、存储、RPC、消息系统
   - 必须显式处理失败路径、超时、资源释放
5. `platform`
   - 封装进程、线程、文件描述符、信号、时钟、指标、日志、环境信息
   - 对上层暴露稳定接口
6. `libs`
   - 优先保持纯净，可独立单测
   - 不依赖进程级生命周期和部署细节

## 模块设计契约

每个可独立交付模块都必须回答：

1. 这个模块负责什么
2. 这个模块不负责什么
3. 这个模块对外公开哪些头文件
4. 这个模块依赖哪些模块
5. 这个模块如何初始化与关闭
6. 这个模块如何暴露错误
7. 这个模块如何验证正确性
8. 这个模块如何被观测

## 头文件与 ABI 边界

1. 公共头文件只暴露稳定接口，不暴露不必要实现细节
2. 非公共头文件放入私有目录，禁止跨模块直接包含
3. 共享库或插件边界必须明确定义 ABI 策略
4. C 接口、插件接口、跨进程协议必须单独审查兼容性

## 生命周期设计

1. 启动阶段
   - 读取配置
   - 验证依赖
   - 初始化日志与指标
   - 初始化线程池或执行器
   - 验证模型或外部依赖
   - 完成预热
2. 运行阶段
   - 提供健康检查
   - 维持资源预算
   - 上报观测信息
   - 处理异常退化
3. 关闭阶段
   - 停止接收新请求
   - 释放工作线程
   - 关闭设备 / 句柄 / socket / 模型上下文
   - 刷新最终日志与指标

## 配置设计

1. 配置必须集中管理
2. 不允许在代码中硬编码环境路径、设备编号、超时、密钥、模型路径
3. 每项配置必须定义：
   - 名称
   - 默认值或“需用户补充”
   - 类型
   - 是否可热更新
   - 风险说明
4. 无效配置必须在启动期失败，不得带病运行

## 错误语义

1. 模块边界必须返回稳定错误语义
2. 业务错误、系统错误、依赖错误、超时、资源耗尽必须可区分
3. 不允许吞错
4. 不允许无上下文返回失败

## 可观测性边界

1. 每个关键模块必须有：
   - 结构化日志
   - 成功 / 失败计数
   - 延迟指标
   - 资源指标
2. AI 模块还必须有：
   - 模型版本
   - 设备信息
   - 推理成功率
   - 超时率
   - 退化路径触发率

## 禁止事项

1. 核心逻辑直接依赖具体模型后端
2. 核心逻辑直接依赖系统信号处理
3. 业务逻辑写在 `main`
4. 平台细节散落在多个服务层模块
5. 模型加载、请求处理、回收逻辑耦合在单个类
6. 仅凭目录名分层，不落实依赖边界
