# Testing and Validation Guide

## 目标

验证必须同时覆盖正确性、稳定性、并发安全、内存安全、性能、安全、部署路径、AI 运行时与回归风险。

## 测试分层

1. 单元测试
   - 面向 `libs`、`services`、小型 `adapters`
   - 快速、确定、独立
2. 集成测试
   - 面向跨模块协作
   - 验证端口与适配器连接
3. 端到端测试
   - 从可执行程序启动到请求完成
   - 验证真实启动、健康检查、配置、退出
4. 压力 / 并发测试
   - 验证队列、线程池、锁、资源预算
5. 基准测试
   - 验证关键路径性能与回归
6. Soak / 长稳测试
   - 验证泄漏、句柄释放、累积退化
7. AI 专项测试
   - 验证模型加载、预热、输入边界、失败退化、版本切换

## 最低验证要求

每次进入主干或发布路径，至少覆盖：

1. 格式化检查
2. 静态检查
3. Debug 构建
4. Release 构建
5. 单元测试
6. 集成测试
7. 端到端 smoke
8. Sanitizer
9. 回归测试
10. 关键路径 benchmark smoke

## Sanitizer 要求

1. ASan：内存越界、UAF
2. UBSan：未定义行为
3. TSan：数据竞争
4. LSan：泄漏检查

规则：

1. 任何 Sanitizer 失败都视为阻断项
2. TSan 对高噪声第三方库的抑制必须审查
3. 不能用“测试环境偶发”跳过 Sanitizer 失败

## AI 专项验证

1. 模型加载成功
2. 模型加载失败
3. 错误模型制品
4. 输入越界
5. 超时
6. 回退路径
7. 版本切换
8. 设备不可用
9. 显存 / 内存预算告警
10. 并发推理安全

## Linux 运行路径验证

1. 启动成功
2. 启动失败
3. 配置错误失败
4. 健康检查通过
5. 信号停止
6. 资源上限触发
7. 权限错误
8. 日志路径 / 数据路径异常
9. 安装后启动 smoke
10. 回滚后恢复 smoke

## 性能验证

1. 指定关键接口的基准测试
2. 记录平均值、P95、P99
3. 测量在 Release 构建下完成
4. 如存在 AI 推理：
   - 冷启动
   - 热启动
   - 单请求
   - 批处理
   - 高并发
5. 性能预算：`需用户补充`

## 交叉验证原则

1. 代码审查不能替代自动化验证
2. 单元测试不能替代集成测试
3. 集成测试不能替代端到端启动路径验证
4. 通过功能测试不能推出线程安全
5. 通过样例输入不能推出边界安全
6. 通过 Debug 构建不能推出 Release 可用

## 回归要求

1. 每个缺陷修复必须新增回归验证
2. 每次复发问题必须回链到 `.harness/error-journal.md`
3. 相同问题第二次出现时，必须补充更强验证
4. 没有验证的修复，不得标记完成

## 验证证据

每次交付至少提供：

1. 执行了哪些构建
2. 执行了哪些测试
3. 哪些检查通过
4. 哪些检查未执行
5. 未执行原因
6. 风险剩余项
