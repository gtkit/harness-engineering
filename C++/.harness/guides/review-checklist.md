# Review Checklist

## 使用方式

审查时先列 Findings，再给开放问题，最后给摘要。所有结论必须可被代码、日志、测试、构建结果、配置或运行证据支撑。

## 严重性定义

- P0：上线阻断 / 数据损坏 / 安全高风险 / 进程不可控 / 结果错误
- P1：合并前必须修复 / 高概率生产故障 / 并发或内存问题
- P2：可排期修复 / 明显质量缺陷 / 测试缺口
- P3：改进项 / 可维护性优化

## 架构审查

- [ ] 模块边界清晰
- [ ] 依赖方向符合架构规范
- [ ] 未把平台细节写入核心库
- [ ] 未把具体 AI Runtime 写入业务编排层
- [ ] 生命周期清晰
- [ ] 关闭路径明确
- [ ] 配置边界清晰
- [ ] 错误语义稳定

## C++ 代码审查

- [ ] 无裸资源泄漏
- [ ] 所有权明确
- [ ] 接口语义清晰
- [ ] 错误处理完整
- [ ] 无吞错
- [ ] 无危险隐式转换
- [ ] 无不必要共享可变状态
- [ ] 无无界缓存或队列

## 并发审查

- [ ] 无脱管线程
- [ ] 锁顺序可解释
- [ ] 无潜在死锁
- [ ] 无竞态共享状态
- [ ] 取消与关闭路径完整
- [ ] 并发测试或 TSan 证据存在

## Linux 系统审查

- [ ] 启动 / 停止路径清晰
- [ ] 信号处理正确
- [ ] 路径与权限未硬编码
- [ ] systemd / 部署假设明确
- [ ] FD / socket 生命周期清楚
- [ ] 资源预算存在
- [ ] 崩溃诊断可追踪

## AI 运行时审查

- [ ] 模型版本可追溯
- [ ] 模型制品可校验
- [ ] 输入输出契约完整
- [ ] 超时与退化策略明确
- [ ] 并发访问策略明确
- [ ] 资源预算明确
- [ ] 加载失败路径已处理
- [ ] 回滚策略存在

## 安全审查

- [ ] 无敏感信息日志泄露
- [ ] 外部输入有边界校验
- [ ] 模型 / 插件 / 动态库加载有校验策略
- [ ] 配置 / 凭据路径安全
- [ ] 依赖来源可追踪
- [ ] 无未审查外部命令执行

## 测试审查

- [ ] 单元测试覆盖关键逻辑
- [ ] 集成测试覆盖关键协作路径
- [ ] 端到端 smoke 存在
- [ ] 回归测试存在
- [ ] Sanitizer 结果存在
- [ ] benchmark 或性能证据存在
- [ ] 未执行项已标注

## 观测与发布审查

- [ ] 结构化日志存在
- [ ] 指标存在
- [ ] 故障诊断信息足够
- [ ] 发布与回滚路径明确
- [ ] 风险项有 owner
- [ ] 历史错误对照检查已执行
