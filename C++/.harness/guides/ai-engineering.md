# AI Engineering Guide

## 适用范围

- 本地模型集成
- 推理服务
- 推理网关
- 模型加载与切换
- CPU / GPU 推理资源控制
- 模型制品校验与发布控制

## 模型制品要求

每个进入生产的模型版本都必须具备可审查清单。至少包含：

1. `model_id`
2. `model_version`
3. `runtime_backend`
4. `target_device`
5. `checksum`
6. `input_contract`
7. `output_contract`
8. `max_batch`
9. `memory_budget`
10. `timeout_budget`
11. `warmup_policy`
12. `fallback_policy`
13. `security_notes`

制品清单格式与存放位置：`需用户补充`

## 模型生命周期

1. 发现
   - 定位制品
   - 读取清单
   - 校验完整性
2. 验证
   - 校验版本
   - 校验 checksum
   - 校验运行时兼容性
   - 校验设备可用性
3. 加载
   - 初始化 runtime
   - 申请资源
   - 构建上下文
4. 预热
   - 执行标准 warmup
   - 记录耗时与资源峰值
5. 服务
   - 接收请求
   - 校验输入
   - 执行推理
   - 返回结果
6. 卸载
   - 回收上下文
   - 回收设备资源
   - 刷新状态

## 输入输出契约

1. 所有外部输入必须做 schema 校验
2. 字符串、张量、图像、音频、文件路径、prompt 长度必须设上限
3. 请求体超限时必须拒绝
4. 输出必须包含足够元数据用于排障
5. 模型输出后处理必须与模型版本关联

## 运行时资源控制

1. 明确 CPU 线程预算
2. 明确内存预算
3. 如存在 GPU，明确显存预算与设备选择策略
4. 批处理必须有上限
5. 队列必须有上限
6. 超时必须可配置
7. 资源不足时必须进入退化或拒绝路径

## 失败与退化

1. 模型加载失败必须阻断对应实例进入服务
2. 推理超时必须记录并返回稳定错误语义
3. 模型损坏、版本不兼容、设备不可用必须可区分
4. 若存在降级模型或 CPU 回退路径，必须显式定义
5. 退化策略未定义时，必须失败而不是隐式切换

## 并发与稳定性

1. 明确模型上下文是否线程安全
2. 明确批处理与并发执行是否可并存
3. 不允许共享不可并发访问的模型上下文
4. 推理线程池、请求队列、回调线程必须有生命周期归属
5. 高并发下必须验证：
   - 死锁
   - 饥饿
   - OOM
   - 排队失控
   - 尾延迟放大

## 版本与发布

1. 模型版本必须可追溯
2. 代码版本与模型版本必须可关联
3. 回滚策略必须明确
4. 同时存在多模型版本时，路由规则必须明确
5. 不允许以覆盖文件方式做无痕升级

## 安全要求

1. 模型制品校验必须进入上线流程
2. 外部下载模型的策略与信任源：`需用户补充`
3. Prompt、输入文本、图像、音频等外部内容必须按不可信输入处理
4. 不允许把用户敏感内容写入调试日志
5. 若支持插件模型 / 自定义算子 / 动态库，必须单独审查风险

## 观测要求

每个模型或运行时适配器至少上报：

1. 模型版本
2. 当前设备
3. 加载耗时
4. 加载失败计数
5. 推理请求量
6. 推理成功率
7. 推理失败率
8. 超时率
9. 平均延迟 / P95 / P99
10. 内存 / 显存峰值

## 测试要求

1. 模型加载成功路径
2. 模型加载失败路径
3. 损坏制品路径
4. 错误输入路径
5. 超时路径
6. 资源上限路径
7. 版本切换路径
8. 回滚路径
9. 长时运行路径
10. 并发路径
