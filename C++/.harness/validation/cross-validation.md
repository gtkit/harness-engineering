# Cross Validation

## 目的

任何“已修复”“线程安全”“无泄漏”“可上线”“性能达标”类结论，必须至少由两类不同证据支撑。

## 结论与证据映射

| 结论 | 最低证据 1 | 最低证据 2 |
| --- | --- | --- |
| 功能正确 | 单元测试 / 集成测试 | 端到端 smoke |
| 无内存越界 / UAF | ASan | 相关回归测试 |
| 无泄漏 | LSan | 长时运行或重复加载卸载测试 |
| 无数据竞争 | TSan | 并发 / 压力测试 |
| 关闭路径正确 | 信号退出测试 | 资源回收日志 / 指标 |
| 配置错误可控 | 启动失败测试 | 审查确认无带病启动 |
| 性能达标 | Release benchmark | 延迟 / 吞吐基线对比 |
| 模型加载可靠 | 加载 smoke | 错误制品失败测试 |
| 退化策略有效 | 设备 / 依赖故障测试 | 观测指标验证 |
| 可发布 | 发布构建通过 | pre-release checklist 通过 |

## 交叉验证规则

1. 静态检查不能替代动态验证
2. 审查不能替代测试
3. 单元测试不能替代系统级启动路径验证
4. 样例输入不能替代边界输入
5. Debug 结果不能替代 Release 结果
6. CPU 路径不能替代 GPU 路径
7. 单次成功不能替代长时稳定性

## 追加验证触发条件

遇到以下情况时，必须补强验证：

1. 同类问题第二次出现
2. 缺陷涉及并发、资源释放、模型版本切换
3. 修复依赖时序变化或关闭路径
4. 修复只在特定构建类型生效
5. 问题来自用户纠正或生产事故回放
